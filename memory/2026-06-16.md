## Key Decisions

1. API route handlers use NextResponse/NextRequest (not NestJS controllers) — Vercel compatibility
2. Vercel API project: Framework Preset = Next.js (bukan NestJS), build = next build, output = .next/
3. Drizzle client `.query` tidak bisa tanpa schema registry — seed scripts pake `db.select().from()` langsung
4. Supabase client type inference rusak untuk `.select('kolom1, kolom2')` — perlu explicit type casting
5. CORS logic extracted ke `apps/api/app/utils/cors.ts` (testable) + thin wrapper di `apps/api/middleware.ts` (Next.js requirement)
6. CORS pure logic extracted ke `packages/db/src/seed/dedupe.ts` (testable) + script wrapper di `dedupe-students.ts` (DB orchestration)
7. **"Most complete wins" dedup strategy** — bukan "keep latest" atau "keep first", tapi score based on field completeness. Rationale: preserves any manual edits (user mungkin udah edit kesan/pesan di salah satu row via admin dashboard), tie-break ke latest createdAt.

## Lesson Learned

- File `vercel.json` override semua settings dashboard — harus dihapus kalo pake framework preset
- Supabase Auth Google OAuth: perlu Client ID + Client Secret, tidak cukup hanya Client ID
- `tsx` tidak auto-load `.env` — perlu explicit env vars passing (`set -a; . ./.env; set +a`)
- API project perlu `next` sebagai dependency untuk route handler types (NextRequest/NextResponse)
- **CRITICAL CORS gotcha**: `app.enableCors()` di NestJS+Express main.ts HANYA berlaku untuk local dev. Production Vercel pakai Next.js route handlers, jadi CORS HARUS dikonfigurasi via Next.js middleware.ts.
- **CORS deployment gotcha**: Vercel deploy dari git, bukan dari working directory. Kalau middleware baru di-create tapi belum di-commit/push, redeploy tidak akan include-nya. Selalu commit + push SEBELUM redeploy untuk trigger CORS fix.
- TDD: pisahkan logic ke testable helper file (`utils/cors.ts` / `dedupe.ts`), middleware/script cuma thin wrapper. Helper bisa di-test dengan jest/vitest biasa tanpa harus setup Next.js/DB infrastructure.
- **Drizzle seed safety**: `onConflictDoNothing()` tanpa target argumen cuma detect conflict di primary key. Kalau PK adalah UUID random (default di `uuid('id').primaryKey().defaultRandom()`), conflict tidak akan pernah terjadi. Untuk safe re-run seed, HARUS specify target: `onConflictDoNothing({ target: [table.nama, table.kelas] })` DAN add UNIQUE constraint di table.
- **TDD pattern untuk dedup logic**: pisah jadi 4 pure functions (score, pick, group, build plan) bukan 1 monolithic function. Tiap function testable independently, dan script tinggal panggil urut. Lebih mudah di-reuse untuk entity lain (teachers, sambutan).
- **`pnpm turbo run test type-check` exit code 1 dari type-check failure di sub-package = cancel semua**. Test pass di sub-package yang lain gak akan report green kalo 1 type-check fail. Untuk verifikasi per-package, run `pnpm --filter @aksana/<pkg> test` individually.
